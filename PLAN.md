--
제품 정의

한 줄 요약: 알림이 발생하면 자동으로 원인을 분석하고, 팀 메신저에 리포트를 전달하는 SRE 자동화 에이전트

핵심 가치: 온콜 엔지니어가 알림을 확인하는 시점에 이미 초기 분석이 완료되어 있음

---

사용자 시나리오

시나리오 1 — 새벽 3시 알림

03:00 Alertmanager: "payment-api 5xx rate > 5%"
↓
03:00 sre-agent: 알림 수신, 자동 분석 시작
03:01 sre-agent → Teams #incident:
"🔔 알림 수신: payment-api 5xx rate > 5%
분석 중..."
03:02 sre-agent: Data Collector → Prometheus, ES 조회
03:03 sre-agent: RCA 완료
03:03 sre-agent → Teams #incident:
"📋 분석 완료 (3분 12초)
원인: DB 커넥션풀 고갈 (신뢰도: HIGH)
즉시 조치: 커넥션풀 사이즈 증가
상세 리포트: [스레드 확인]"
↓
03:15 온콜 엔지니어 기상, Teams 확인
→ 이미 원인과 조치방안이 있음
→ 스레드에서 "@sre-agent DB 슬로우쿼리도 확인해줘" 후속 질문
03:16 sre-agent: 추가 분석 후 스레드에 응답
↓
03:30 엔지니어: 조치 완료, 리액션으로 "resolved" 표시
03:30 sre-agent: 인시던트 KB에 자동 저장
→ 다음에 비슷한 알림 오면 "지난 4/8에 같은 패턴 발생,
그때는 DB 커넥션풀이 원인" 자동 참조

시나리오 2 — 노이즈 알림

14:00 Alertmanager: "node-exporter CPU > 80%" (1분간 스파이크)
↓
14:00 sre-agent: 알림 수신, 심각도 판단
14:01 sre-agent: query_instant로 빠른 확인 → 이미 정상 복귀
14:01 sre-agent → Teams #alerts:

→ 풀 RCA 안 돌림. 토큰/시간 절약.

09:00 교대 시간
↓
09:00 sre-agent → Teams #oncall:
"📊 지난 12시간 온콜 요약 - 알림 7건 (critical 1, warning 3, resolved 3) - 미해결: payment-api 지연 (03:00~, 조치 진행 중) - 주요 변경: deploy v2.3.1 (02:45)
상세: [스레드]"

알림 → 분석 파이프라인 설계

┌─────────────┐ ┌──────────────┐ ┌──────────────┐ ┌─────────────┐
│ Alert Source │────→│ Intake │────→│ Analyzer │────→│ Delivery │
│ │ │ │ │ │ │ │
│ Alertmanager │ │ 수신 │ │ 분석 수행 │ │ 결과 전달 │
│ PagerDuty │ │ 중복 제거 │ │ KB 참조 │ │ KB 저장 │
│ Grafana │ │ 심각도 분류 │ │ 비용 관리 │ │ 후속 대화 │
│ Datadog │ │ 그룹핑 │ │ │ │ │
└─────────────┘ └──────────────┘ └──────────────┘ └─────────────┘

Stage 1 — Intake (수신 + 전처리)

여기가 현재 코드에 가장 빠져있는 부분입니다.

중복 제거: 같은 알림이 1분 간격으로 3번 오면 한 번만 분석
key = (alertname, service, instance)
if key in recent_alerts (within 5min):
skip or append to existing analysis

심각도 분류: 모든 알림에 풀 RCA를 돌리면 비용이 폭발
critical → 즉시 풀 분석 (4 agents 전부)
warning → 경량 분석 (data_collector만, 이상 있으면 풀 분석으로 승격)
info → 로그만 (메신저에 요약 한 줄)
resolved → 요약 ("14:00~14:23 지속, 자동 복구")

그룹핑: 동시에 터진 연관 알림을 하나의 인시던트로 묶음
14:00 payment-api 5xx > 5%
14:01 payment-db connection_pool exhausted
14:01 payment-api latency p99 > 2s
→ 하나의 인시던트로 묶어서 한 번만 분석

Stage 2 — Analyzer (분석)

현재 Core Engine이 이미 담당. 추가할 것:

KB 참조: 분석 시작 전에 과거 유사 인시던트 조회
분석 시작
→ KB search("payment-api 5xx connection pool")
→ 유사 인시던트 발견: INC-1712345 (2주 전, DB 커넥션풀, 신뢰도 HIGH)
→ Orchestrator에게 컨텍스트로 전달
→ "지난번과 같은 패턴인지 우선 확인" → 분석 시간 단축

비용 관리: 월 API 비용 cap 설정
cost_management:
monthly_budget_usd: 100
per_analysis_estimate_usd: 0.5 # 대략적 추정
alert_at_percent: 80
action_at_limit: "lightweight_only" # 예산 소진 시 경량 분석만

Stage 3 — Delivery (전달)

이전에 논의한 Messenger Protocol 활용:

실시간 진행 상태:
Teams 스레드:
🔔 알림: payment-api 5xx rate > 5%
⏳ 분석 중...
→ Data Collector (Prometheus, ES 조회 중) ← 실시간 업데이트
→ Root Cause Analysis 진행 중...
✅ 분석 완료 — 아래 리포트 확인

    📋 [최종 리포트 메시지]

후속 대화: 같은 스레드에서 follow-up
엔지니어: "@sre-agent 이거 롤백해야 해?"
sre-agent: "현재 에러율이 감소 추세(5% → 2.1%)입니다.
10분 더 모니터링 후 1% 이하로 안 떨어지면
v2.3.0으로 롤백을 권장합니다."

인시던트 종료 + KB 저장:
엔지니어: 이모지 ✅ 또는 "@sre-agent resolved"
sre-agent:
→ KB에 인시던트 저장 (컨텍스트, RCA, 조치, 소요시간)
→ "인시던트 저장 완료 (INC-1712456). 포스트모템 드래프트가 필요하면 말씀하세요."

구현 페이즈

Phase 1 — 파이프라인 뼈대 

현재 코드를 리팩터링하여 파이프라인 구조로 전환

목표: Alertmanager → 자동 분석 → 콘솔 출력 (메신저 없이)

작업:

- intake/ 모듈: 중복 제거, 심각도 분류, 그룹핑
- analyzer/ 모듈: 기존 Core Engine 래핑 + KB 참조
- delivery/ 모듈: 콘솔 출력 (메신저 전 단계 검증용)
- 기존 webhook.py를 intake로 리팩터링

검증: docker-compose로 Prometheus + Alertmanager 띄우고
테스트 알림 발사 → 자동 분석 → 콘솔에 리포트 출력

Phase 2 — 메신저 연동 

목표: 분석 결과가 Teams/Teams에 자동으로 전달 (Teams Webhook)

작업:

- messenger/ 모듈: Protocol + Adapter 패턴
- TeamsAdapter 구현 (Teams-bolt)
- 실시간 진행 상태 (스레드 메시지 업데이트)
- 스레드에서 follow-up 대화

검증: 알림 → Teams 스레드에 리포트 + follow-up 대화

Phase 3 — 지능화 (2주)

목표: 분석이 점점 더 빨라지고 정확해짐

작업:

- KB 자동 저장/참조 파이프라인
- 유사 인시던트 자동 매칭 → "이전에도 같은 패턴"
- 비용 관리 (budget tracking, severity 기반 분석 수준 조절)
- 온콜 핸드오프 요약 (스케줄 기반)

검증: 같은 유형의 알림을 2회 발생시켜 2번째에 KB 참조 확인

Phase 4 — 확장 (이후)
- 포스트모템 자동 드래프트
- Grafana 대시보드 링크 자동 첨부
- 비개발자용 Web UI (인시던트 히스토리 열람)

---
  먼저 답: Teams 전달은 에이전트가 아닌 코드 로직

  에이전트가 할 일: 생각하기, 판단하기, 데이터 해석하기 (LLM 필요)
  코드가 할 일:     메시지 보내기, 버튼 처리, 중복 제거, 라우팅 (결정론적)

  Teams에 메시지 보내는 건 판단이 필요 없는 확정적 동작이므로 코드 로직입니다. 반면 "이 알림을 어떻게 분석할까"는 에이전트의 일입니다.
